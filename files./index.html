// what is flashbot?

Python CLI that automates the embedded firmware workflow for Arduino boards.

// the problem Plug in board → IDE → select port → compile → upload → serial monitor → verify → repeat. FlashBot collapses steps 2–7 into one action.
[ 01 ]

Detection

Listens for USB hotplug events at the OS level. Sees your board appear on /dev/ttyACM0 and reads its hardware ID.

pyudev · hotplug listener
[ 02 ]

Identification

Checks internal board database. Known board → loads FQBN. Unknown → shows USB IDs and asks once. Remembered forever.

BOARD_TABLE · user_boards.json
[ 03 ]

Sketch selection

Scans sketches/ folder tree for firmware projects matching your board. Auto-selects if one exists, otherwise shows a numbered menu.

firmware_manifest.json
[ 04 ]

Compilation

Calls arduino-cli to compile. Computes SHA-256 hash of source — if unchanged since last success, skips compilation entirely.

hash-guarded builds
[ 05 ]

Flashing

Uploads the compiled binary over USB/serial. Factory pattern in flasher/ means adding a board family = one file + one line.

arduino-cli · esptool (stub)
[ 06 ]

Testing & logging

Opens serial connection, runs test suite (blink → port opens, echo → sends HELLO, expects HELLO). Results saved to SQLite.

pyserial · SQLite

// what happens step by step

1. Detection — USB hotplug event on /dev/ttyACM0.
2. Identification — Board database matches USB IDs to Arduino Uno WiFi R4.
3. Sketch — Scans sketches/arduino/renesas_uno/unor4wifi/ → auto-selects the only sketch.
4. Compile — SHA-256 match → sketch unchanged, compile skipped.
5. Flash — arduino-cli uploads over serial.
6. Test — Serial connection opens, test suite passes.
7. Log — Result saved to logs/flashbot.db.

$ flashbot — key design decisions

decisionrationale
Factory pattern in flasher/ Adding a board family = 1 file + 1 line. No changes to main orchestration.
Hash-guarded builds SHA-256 of .ino stored in DB. Skips compile if unchanged since last success.
Unknown board flow Prompts for name + FQBN, persists to JSON. Survives restarts across sessions.
Sketch folder tree sketches/<vendor>/<arch>/<board>/ auto-maps to FQBN. Manifest regenerates on every startup.
Rich CLI + plain fallback Rich for interactive use; main.py works when Rich is unavailable or for CI piping.
Per-sketch test suites Each sketch defines its own test protocol. Blink → port opens. Echo → send/verify.

$ flashbot — troubleshooting

"No sketches configured for [board name]"

Your sketches/ folder doesn't have a path matching that FQBN. Check config/firmware_manifest.json to see what was detected.

"Compiled sketch not found in ..."

The build cache path and upload path got out of sync. Make sure flasher/arduino_flasher.py has the self.build_dir logic with --input-dir passed to upload.

"No flasher registered for FQBN"

The board is recognized but has no flasher class. Currently only arduino: FQBNs have full flashers.

Board prompts again after flash

Arduino boards reset after upload and briefly re-enumerate on USB. FlashBot has a 5-second debounce cooldown.

"ImportError: cannot import name 'ResultStore'"

You have an old result_store.py. The current codebase uses plain functions (save_result, get_results).
